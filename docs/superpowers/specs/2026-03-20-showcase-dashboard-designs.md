# PKM Showcase Dashboard Designs

Date: 2026-03-20

## Goal

Capture the settled first-slice showcase designs for `vimwiki-query` before
implementation begins.

This document complements the dedicated weekly-review spec:

- `docs/superpowers/specs/2026-03-20-weekly-review-showcase-design.md`

The intent is to lock down scope and query behavior for the remaining showcase
dashboards so implementation can proceed later without reopening design
questions.

## First-Slice Showcase Set

The first implementation slice includes these showcase queries:

- Weekly review
- Project health dashboard
- One-project task rollup companion query
- Waiting / blocked queue
- Meeting action-item inbox
- Personal CRM dashboard
- Research / source inbox

Explicitly out of scope for this slice:

- unlinked-mention cleanup

The first slice should stay explicitly link-based. It should not attempt
plain-text entity detection or inferred relationships.

## Shared Query Conventions

These designs share a few conventions:

- showcase queries are shell scripts under `showcase/queries/`
- the scanner remains unchanged unless a later design explicitly requires it
- query logic should be expressed with existing scanner output and `jq`
- first-slice output should stay text-first and operational rather than trying
  to become a general reporting framework
- project, person, meeting, and source are query-layer conventions based on
  `frontmatter.type`

## Weekly Review

Weekly review already has a dedicated design doc. The settled scope is:

- one markdown report generated by one showcase script
- parameterized by `WEEK=YYYY-Www`
- sections ordered by actionability rather than by source note type
- includes:
  - open diary tasks
  - due-this-week tasks from all notes
  - waiting / blocked tasks
  - active projects not updated this week
  - project mentions
  - people mentions
  - done-this-week diary tasks
  - updated non-diary notes
- empty sections should still render with reassuring text

Implementation should follow the dedicated weekly-review spec rather than this
summary.

## Project Health Dashboard

### Goal

Provide one compact dashboard that lets a weekly-review-style workflow quickly
see which projects need attention.

### Scope

Included projects:

- pages where `frontmatter.type == "project"`
- exclude pages where `frontmatter.status == "archived"`

The dashboard is parameterized by:

```sh
TODAY=YYYY-MM-DD
```

### Output Shape

The dashboard should emit compact one-line bullets, not a markdown table.

Each project line should show:

- project link
- status
- open task count
- due-soon count
- waiting/blocked count
- diary mention count
- last updated date

Zero-signal projects should still appear.

### Counting Rules

Open task count includes:

- open tasks on the project page itself
- open diary tasks that mention that project explicitly

Waiting/blocked count includes:

- open tasks on the project page itself
- open diary tasks that mention that project explicitly and include waiting
  state tags

Diary mention count means:

- diary task records that mention the project explicitly

Due-soon means:

- open tasks with explicit `due`
- due within the next 7 days from `TODAY`

Last updated means:

- raw date derived from the project page `mtime`

### Ordering

Projects should be ordered by:

1. waiting/blocked count descending
2. due-soon count descending
3. open task count descending
4. last updated ascending
5. path alphabetical

### Companion Query

This dashboard requires a separate first-class companion query:

- one-project task rollup

The companion query should split results into:

- tasks on the project page
- diary tasks mentioning that project

This companion should be documented in the cookbook and showcase as its own
entry, not buried as an implementation helper.

## Waiting / Blocked Queue

### Goal

Provide one global queue of stalled work that can be skimmed quickly during
review.

### Scope

- parameterless in the first slice
- all note types
- open tasks only

### Classification Rules

Task tags map to queue sections as follows:

- `wait`, `waiting`, `waitingfor` => `Waiting`
- `blocked` => `Blocked`

If both waiting and blocked signals appear on the same task, the task belongs
in `Blocked`.

### Output Shape

The query should emit two flat sections:

- `Waiting`
- `Blocked`

Task lines should show:

- source note link
- task text

Existing project or people links already present in the task text should remain
visible in the output.

The first slice should not group by blocker target or inferred dependency.

## Meeting Action-Item Inbox

### Goal

Surface open action items from meeting notes without requiring a special folder
layout.

### Scope

A meeting note is any page where:

- `frontmatter.type == "meeting"`

There is no folder restriction in the first slice.

Only open tasks should be shown. Meetings with zero open tasks should be
omitted.

### Output Shape

Primary output is grouped by meeting note.

Each meeting header should show:

- meeting note link
- optional `with`
- optional `project`

`with` and `project` frontmatter may be either:

- single values
- or lists

### Ordering

Meetings should be sorted most recent first.

Date ordering should prefer:

- frontmatter `date` when present

and otherwise fall back to:

- path or filename ordering

## Personal CRM Dashboard

### Goal

Provide one compact dashboard of people who need follow-up or whose records are
stale.

### Scope

A person record is any page where:

- `frontmatter.type == "person"`

Exclude pages where:

- `frontmatter.status == "archived"`

The first slice is parameterless.

### Output Shape

Each person should render as a compact bullet showing:

- person link
- status
- open follow-up count
- diary task mention count
- explicit `last_contact` if present
- last updated date

Only explicit links count. The query should not use raw-name matching.

The first slice should not infer `last_contact` from diary mentions.

### Ordering

People should be ordered by:

1. open follow-up count descending
2. missing `last_contact` first
3. oldest `last_contact` first
4. oldest `last updated` first
5. path alphabetical

## Research / Source Inbox

### Goal

Provide an actionable inbox of source notes that still need ingestion work.

### Scope

A source record is any page where:

- `frontmatter.type == "source"`

The primary first-slice workflow is:

- sources needing ingestion

Recommended status model:

- `inbox`
- `reading`
- `processed`
- `archived`

The first-slice actionable set is:

- non-archived sources whose status is not `processed`

### Output Shape

Output should be grouped by status in this order:

1. `inbox`
2. `reading`

`processed` and `archived` should be omitted in the first slice.

Within each group, sources should be ordered by oldest `mtime` first.

Each source line should show:

- source note link
- status
- topic if present
- raw `url` if present

The first slice should stay minimal. It should not add author, year, project
linkage, or other secondary metadata yet.

## Deferred Showcase

### Unlinked-Mention Cleanup

This showcase remains deferred and should not be part of the first
implementation slice.

Reasoning:

- the first slice should rely on explicit links only
- plain-text entity detection introduces noise and a larger design surface
- the more operational dashboards above are higher value and already well
  specified

When this showcase is revisited later, it should be treated as a separate
design exercise rather than quietly folded into the current slice.
